## 2025-02-13 - [Missing Address Constraint on Treasury Account]
**Vulnerability:** The Anchor smart contract (`programs/casino_solana/src/lib.rs`) failed to enforce an `address` constraint on the `tesoreria` account in the `ComprarBloque` instruction. This meant an attacker could supply an arbitrary public key as the treasury and successfully invoke the instruction, bypassing the intended payment to the true protocol treasury.
**Learning:** In Solana smart contracts, accounts provided to an instruction must be explicitly validated. If an account is meant to represent a specific, constant destination for funds (like a protocol treasury), it must have an `address` constraint using a constant or the `pubkey!` macro, or its address must be verified manually inside the instruction body. Without this, attackers can siphon funds or bypass fees.
**Prevention:** Always use the `#[account(address = ...)]` constraint for static, known protocol accounts. Ensure that the corresponding public key used in the frontend strictly matches this constraint to prevent unintended instruction failures for legitimate users. Also, avoid using placeholder IDs like the System Program ID as dummies for constraints; always use validly generated keypairs.
