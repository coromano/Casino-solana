## 2025-02-14 - [CRITICAL] Missing Address Constraints on Anchor Accounts
**Vulnerability:** The Anchor smart contract allowed users to provide arbitrary accounts for treasury deposits (via the `tesoreria` account in `ComprarBloque`) because it lacked an `address` constraint validation. A malicious user could supply their own address, successfully invoking the contract and effectively stealing or diverting funds intended for the treasury.
**Learning:** In Solana smart contracts, accounts that are constant destinations for protocol funds (like a protocol treasury or a fee collection account) must explicitly validate their address using an `address` constraint (e.g., using `pubkey!("...")` macro from `solana-program`) to ensure they match the hardcoded, intended destination.
**Prevention:** Always use `#[account(mut, address = pubkey!("EXPECTED_ADDRESS"))]` for constant addresses representing treasury, admin, or fee wallets in Anchor constraints to prevent arbitrary account replacement by attackers. Ensure `solana-program = "=2.3.0"` is added in `Cargo.toml` if using `anchor-lang = "0.32.1"` to prevent pubkey type mismatches.
